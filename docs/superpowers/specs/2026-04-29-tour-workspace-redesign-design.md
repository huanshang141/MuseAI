# MuseAI `/tour` 工作台化重构设计文档

> 日期: 2026-04-29
> 状态: Draft（用户需求已确认，可进入计划阶段）
> 范围: `frontend/` 中 `/tour` 页面及其相关布局组件
> 关联文档: `docs/superpowers/specs/2026-04-21-frontend-museum-redesign-design.md`

---

## 1. 背景与目标

### 1.1 当前问题

`/tour` 页面当前仍采用“独占全屏 + 自定义深色壳层 + 隐藏全站导航”的旧模式，与现有前端设计系统和信息架构出现显著脱节：

1. 与全站布局割裂: `AppHeader` 和全站导航在 `/tour` 被完全隐藏，用户无法快速跨页切换
2. 与设计系统割裂: `/tour` 多处硬编码颜色、间距和动效，未充分复用 tokens 与 Museum 组件
3. 功能密度不足: 正式导览阶段仅有线性聊天与“下一个展品”，缺少可切换的工作台能力
4. 可扩展性弱: 缺少“设置层”和“上下文快捷入口”，难以平滑承载风格参数、TTS 等后续能力
5. 移动端体验不完整: 正式导览阶段尚未形成“标签 + 工具抽屉 + 输入粘底”的统一策略

### 1.2 本次重构目标

在不改后端协议与核心导览状态机的前提下，完成 `/tour` 的工作台化改造：

1. `/tour` 不再独占全屏，恢复全站顶部导航标签
2. 仅在 `tourStep = 'tour'` 阶段启用“二级标签 + 左侧功能区”工作台
3. 四个二级标签完整落地: `导览会话 / 展品速览 / 导览进度 / 导览设置`
4. 左侧区域采用混合模式: 上半区为四标签快捷入口，下半区为常驻工具
5. 会话区交互形态对齐智能问答页: 消息区 + 输入区 + 流式反馈 + 快捷提问
6. 展品速览支持模板菜单补全输入: 点击展品 -> 选择模板 -> 自动填入输入框（不自动发送）
7. 导览设置支持前端偏好 + 对话风格参数，并预留 TTS 设置占位
8. 视觉风格升级为“浅色主基调 + 局部深色沉浸区”，与全站 harmonize
9. 手机端完整适配（无横向滚动、输入不被键盘遮挡、工具区抽屉化）

### 1.3 非目标

1. 不改 `useTour` 的导览主状态机流程（`onboarding -> opening -> hall-select -> tour -> report`）
2. 不引入导览多会话管理（仍保持单导览会话机制）
3. 不改导览后端 API 协议或新增后端字段
4. 不实现真实 TTS 能力（仅前端设置占位）
5. 不重构首页智能问答业务逻辑

## 2. 需求确认结果（与用户对齐）

本设计基于用户逐项确认，关键约束如下：

1. 恢复全站顶部导航标签
2. 在 `/tour` 内增加二级标签，并提供左侧功能标签区
3. 二级标签数量固定为 4 个：
   - 导览会话
   - 展品速览
   - 导览进度
   - 导览设置
4. 二级标签与左侧功能区仅在 `tourStep='tour'` 显示
5. `onboarding/opening/hall-select/report` 继续保留沉浸单页流程
6. 导览设置包含:
   - 前端偏好
   - 对话风格参数（回答长短/讲解深浅/术语难度，前端提示词注入）
   - TTS 设置占位（音色等，仅占位）
7. 会话形态改造深度: 仅对齐交互形态，不做导览多会话
8. 展品速览点击行为: 弹出模板菜单，选择后填入输入框，不自动发送
9. 视觉基调: 浅色为主 + 局部深色沉浸区
10. 左侧功能区结构: 混合模式（上半区标签快捷、下半区常驻工具）

## 3. 方案对比与选型

### 3.1 候选方案

1. 方案A: 在现有 `ExhibitTour/ExhibitChat` 局部拼接工作台壳层
2. 方案B: 新增 `TourWorkspace` 容器，分层重构 `tourStep='tour'` 阶段
3. 方案C: `/tour` 拆分多子路由页面

### 3.2 最终选型

采用 **方案B（推荐）**。

原因：

1. 结构清晰，能在保留 `useTour` 业务流程的前提下引入可维护工作台
2. 与设计系统契合度最高，便于实现“浅色主基调 + 局部沉浸”
3. 能自然承载“模板补全”“风格注入”“TTS占位”等可扩展能力
4. 相比子路由拆分风险更低，不会引入额外路由状态复杂度

## 4. 信息架构与页面结构

### 4.1 `/tour` 双态结构

`/tour` 同一路由内按 `tourStep` 分两种展示态：

1. 沉浸流程态（不变）:
   - `onboarding`
   - `opening`
   - `hall-select`
   - `report`
2. 导览工作台态（新增）:
   - `tourStep === 'tour'`

### 4.2 工作台态布局层级

```
App Shell（保留全站 Header）
└── TourView
    ├── 非 tour 步骤：沿用现有沉浸单页组件
    └── tour 步骤：TourWorkspace
        ├── 左侧功能区（快捷标签 + 常驻工具）
        └── 右侧主内容区
            ├── 二级标签栏
            └── 标签内容面板
                ├── 导览会话
                ├── 展品速览
                ├── 导览进度
                └── 导览设置
```

### 4.3 左侧功能区（混合模式）

上半区：标签快捷入口（与主标签同步）

1. 导览会话
2. 展品速览
3. 导览进度
4. 导览设置

下半区：常驻工具

1. 当前展厅信息
2. 当前展品信息
3. 展品总进度摘要
4. 快捷模板入口（与展品速览联动）
5. 快捷开关入口（如自动滚动、快捷提示词显示）

## 5. 四个二级标签详细设计

### 5.1 导览会话（主工作区）

目标：在单会话机制下，对齐智能问答页的核心交互形态。

功能组成：

1. 会话状态条
   - 当前展厅
   - 当前展品
   - 导览人设标签
   - 发送状态（空闲/思考/流式生成）
2. 消息流区域
   - 用户消息与导览员消息分离样式
   - 流式输出实时渲染
   - 自动滚动可配置
3. 快捷提问区
   - 常用问题 chips
   - 可被设置开关隐藏
4. 输入区
   - 文本输入
   - 发送按钮
   - 草稿保持（可关闭）

交互原则：

1. 不改变 `useTour.sendTourMessage` 的基础协议
2. 输入草稿与模板填充共用同一草稿状态
3. 标签切换不清空草稿与消息滚动位置

### 5.2 展品速览

目标：提供“展品列表 + 模板补全”高效提问入口。

功能组成：

1. 当前展厅展品列表
2. 展品卡片（名称、简述、序号、是否当前展品）
3. 每个展品可触发模板菜单

模板菜单首批：

1. `介绍一下{{展品名}}`
2. `{{展品名}}有什么历史争议？`
3. `{{展品名}}和其他展品有什么联系？`

模板菜单行为：

1. 选择模板后，自动切换到 `导览会话` 标签
2. 把模板文本填入输入框
3. 输入框自动聚焦
4. 不自动发送

### 5.3 导览进度

目标：把线性导览状态可视化，提供必要行动入口。

功能组成：

1. 展厅进度卡
   - 当前展厅名称
   - 已浏览展品数 / 总展品数
2. 当前展品卡
   - 当前序号
   - 当前展品名称
   - 下一个展品提示
3. 行为摘要
   - 已进入展厅
   - 深挖次数
   - 提问次数
4. 行动按钮
   - 下一个展品
   - 完成本展厅
   - 返回选厅（当状态允许）

### 5.4 导览设置

目标：让用户在不改后端的情况下调节体验，并预埋 TTS 配置位。

A. 前端偏好

1. 字体大小：小/中/大
2. 消息密度：紧凑/舒展
3. 自动滚动：开关
4. 快捷提问显示：开关
5. 草稿记忆：开关

B. 对话风格参数（提示词注入）

1. 回答长短：简短/适中/详细
2. 讲解深浅：入门/标准/深入
3. 术语难度：通俗/专业/学术

C. TTS 设置（占位，不可用态）

1. 音色（占位选择）
2. 语速（占位选择）
3. 自动朗读（占位开关）
4. 上线提示文案

## 6. 视觉与交互规范

### 6.1 视觉总策略

1. 工作台整体采用全站浅色 token
2. 会话消息区使用局部深色沉浸容器
3. 通过分割线、纹样和间距保持博物馆风统一
4. 避免再次出现“整页独立换肤”造成割裂

### 6.2 Token 使用约束

1. 仅使用 design-system token 变量
2. 禁止新增硬编码品牌色（特殊状态色除外）
3. 消息气泡、输入区、标签栏必须可在浅色基础上清晰对比

### 6.3 局部沉浸区定义

推荐仅对以下区域使用 dark-surface：

1. 导览会话消息流容器背景
2. AI 回复气泡的纹样底层
3. 流式生成中光标与弱强调色

### 6.4 动效与反馈

1. 标签切换动效 <= 200ms
2. 模板填充后提供轻量 toast/inline 提示
3. 设置项变更即时生效并可撤销（恢复默认）

## 7. 状态管理与数据流

### 7.1 现有状态保持

继续以 `useTour` 作为导览业务单一事实源，保留：

1. `tourStep`
2. `currentHall`
3. `currentExhibit`
4. `exhibitIndex`
5. `chatMessages`
6. `streamingContent`
7. `sendTourMessage()`
8. `completeHall()`

### 7.2 新增工作台状态层

新增 `useTourWorkbench` 管理纯前端视图/偏好状态：

1. `activeTab`
2. `chatDraft`
3. `uiPreferences`
4. `stylePreferences`
5. `ttsPlaceholderPreferences`
6. `insertTemplateForExhibit(exhibit, templateKey)`
7. `buildStyledPrompt(rawInput, stylePreferences)`

### 7.3 提示词注入策略

发送消息时进行前端拼装：

1. 用户可见输入保持原文
2. 内部发送文本可在前缀加入风格指令
3. 风格指令仅来自设置页参数
4. 可通过开关关闭注入并回退原文发送

示意：

```
[风格约束前缀]
回答长度: 适中
讲解深浅: 标准
术语难度: 通俗
---
用户问题正文
```

### 7.4 本地持久化

1. 偏好设置落地 localStorage
2. 草稿记忆按 `tourSession.id` 隔离
3. 读取异常时回退默认值，不阻塞页面

## 8. 响应式与移动端设计

### 8.1 桌面端（>=1024）

1. 左侧功能区固定宽度约 280px
2. 二级标签常驻顶部
3. 会话区输入框底部粘附

### 8.2 平板端（768~1023）

1. 左侧功能区收窄至 220~240px
2. 二级标签允许横向滚动
3. 内容卡片布局适当压缩

### 8.3 手机端（<768）

1. 左侧功能区改为抽屉（Drawer）
2. 工具按钮置于工作台头部
3. 二级标签栏 sticky + 横向滚动
4. 会话输入区 sticky 到底部安全区
5. 展品速览卡片单列
6. 模板菜单优先使用可触达面积更大的弹层样式

### 8.4 移动端必须满足的验收条件

1. 不出现横向滚动
2. 键盘弹起后输入区仍可见
3. 切换标签不丢输入草稿
4. 抽屉开关不重置当前标签和滚动

## 10. 可用性与异常处理

1. 展品列表加载失败
   - 显示错误状态与重试按钮
2. 流式中断
   - 保留已生成内容
   - 提供重试操作
3. 当前展厅无展品
   - 给出返回选厅 CTA
4. 设置读取异常
   - 自动回退默认值
   - 弹轻量告警
5. 模板填充失败（空展品/模板缺失）
   - 不切换标签
   - 显示错误提示

1. 展品模板菜单选择后填充输入框
2. 模板填充后自动切换到 `导览会话`
3. 移动端抽屉开关逻辑
4. 标签切换保持草稿

### 11.3 手工回归

分辨率三档：1280 / 768 / 390

关键路径：

1. `onboarding -> opening -> hall-select -> tour`
2. 工作台四标签功能可用
3. 展品模板补全可用
4. 风格参数变更后发送行为正确
5. `tour -> report` 无回归

## 12. 里程碑与实施顺序

1. 里程碑M1: 壳层改造（恢复顶部导航 + tour阶段工作台入口）
2. 里程碑M2: 四标签与左侧混合功能区落地

每个里程碑均要求：

1. 先补测试，再改实现
2. 桌面与移动端各做一次可用性验证
3. 完成后执行 `vitest` 相关用例

## 13. 验收标准（Definition of Done）

1. `/tour` 不再隐藏全站顶部导航
2. `tourStep='tour'` 出现完整工作台（四标签 + 左侧混合功能区）
3. 非 `tour` 步骤仍保持沉浸单页
4. 展品模板菜单可把文本填入输入框并跳转会话标签
5. 风格设置可通过前端提示词注入影响发送内容
6. TTS 设置区已占位且交互清晰标注“未上线”
7. 手机端无横向滚动，输入区可用
8. 现有导览流程和报告生成无功能回归
9. 新增测试覆盖关键交互与核心状态逻辑
